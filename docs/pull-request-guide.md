# Pull Request 작성 기준

PR 본문은 리뷰어가 변경 이유와 영향 범위를 빠르게 판단할 수 있게 쓰는 문서다. 빈 템플릿 흔적이나 자동 생성 요약이 섞이면 팀원이 실제로 봐야 할 내용이 흐려지므로 아래 기준을 따른다.

## 기본 형식

```md
## 작업 내용
- 이번 PR에서 실제로 바꾼 내용을 적는다.

## 변경 배경
- 왜 이 변경이 필요한지 적는다.

## 공유 사항
- FE, 배포, env, API 명세, 운영 방식에 영향을 주는 내용을 적는다.
- 없으면 `없음`이라고 적는다.

## 확인한 내용
- 실행한 명령어 자체보다 확인한 결과를 적는다.
```

## 작성 규칙

- 관련 이슈가 없으면 `close: #` 같은 placeholder 섹션을 만들지 않는다.
- `테스트 방법` 섹션을 만들지 않는다. 명령어 나열 대신 어떤 동작이 검증됐는지 `확인한 내용`에 적는다.
- CodeRabbit Summary/Release Notes 같은 자동 생성 문구를 PR 본문에 붙이지 않는다.
- PR 제목은 커밋 컨벤션과 맞춰 `feat: ...`, `fix: ...`, `docs: ...` 형식을 사용한다.
- FE가 알아야 하는 응답 필드, 에러 코드, env, 배포 시점 차이는 `공유 사항`에 적는다.

## 예시

```md
## 작업 내용
- `profileCompleted=false` 사용자가 보호 API를 직접 호출하면 `403 AUTH_REQUIRED_PROFILE_INFO`를 반환하도록 guard를 추가했습니다.
- 프로필 조회와 프로필 완료 API는 guard 예외로 유지했습니다.

## 변경 배경
- FE가 CB-02로 라우팅하더라도 토큰을 가진 사용자가 보호 API를 직접 호출할 수 있어 서버 차단이 필요했습니다.

## 공유 사항
- FE는 로그인 응답 또는 `GET /api/users/me`에서 `profileCompleted=false`이면 CB-02로 이동시키면 됩니다.
- 보호 API에서 `AUTH_REQUIRED_PROFILE_INFO`가 오면 같은 화면으로 유도하면 됩니다.

## 확인한 내용
- 프로필 미완료 사용자는 보호 API에서 차단되는 것을 확인했습니다.
- 프로필 미완료 사용자도 내 프로필 조회와 프로필 완료 API는 호출 가능한 것을 확인했습니다.
- 전체 테스트가 통과하는 것을 확인했습니다.
```
